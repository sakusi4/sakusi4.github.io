본 글에서는 포인터의 기본 문법을 다루지 않습니다.

포인터(Pointer)
---
포인터는 메모리 주소를 저장하는 변수다. 포인터를 어렵다고 여기는 사람들이 종종 있는데, 책에서 포인터 이전의 내용을 겉핥기로 배웠기 때문이다.

요컨대 변수를 공부하면서 상자에 값을 집어넣는다는 정도의 수준으로 공부했다면 포인터를 접했을 때 어렵게 느낄 수 있다. 글이 좀 길지만, 차분히 읽어보길 바란다.

포인터는 왜 필요할까?
---

{% highlight c linenos=table %}
// Example 1
void increase(int b)
{
    b++;
}

int main()
{        
    int a = 0;
    increase(a);
}
{% endhighlight %}

작성자의 의도는 main 함수에 있는 a 변수의 값을 increase 함수로 증가시키는 것이다. 하지만 increase 함수의 변수 b는 main 함수의 변수 a와 다른 공간에 할당되어있다.
쉽게 말해 a와 b는 조금의 관계도 없다. a의 값이 b의 값으로 **복사** 됐을 뿐이다. 그러므로 b를 이용해서 절대 main 함수의 a를 수정할 수 없다.

옆집의 컴퓨터와 똑같은 컴퓨터를 산 다음, 이것을 부숴도 옆집의 컴퓨터에는 아무런 영향이 없다. 
이제 일본 공포 영화에 등장하는 지푸라기 저주 인형의 원리도 이해할 수 있다. 알고보니 저주 대상의 포인터였던 것이다.

Example 2는 Example 1의 원래 의도를 포인터로 구현한다.

{% highlight c linenos=table %}
// Example 2
void increase(int *b)
{
    (*b)++;
}

int main()
{        
    int a = 0;
    increase(&a);    
}
{% endhighlight %}

&연산자를 통해 얻은 a의 주소 값을 b 포인터에 **복사** 한다. Example 1과 같다.

이제는 *기호를 통해 간접적으로 a의 메모리에 접근하여 수정할 수 있다. increase 함수가 종료되면 b 변수는 삭제된다. a에는 영향이 없다.
친구의 집 주소를 까먹었다고 친구의 집이 사라지지 않는 것과 같다. 

여기까지 읽고 나면 책에서 봤던 call-by-value, call-by-reference가 자연스레 떠오른다.
분명히 말하지만, call-by-reference는 없다. Example 1과 Example 2는 아무런 차이가 없다. 하지만 여전히 호출 부의 &기호와 매개변수의 *기호가 마음에 걸린다.

Call-by-reference?
---

{% highlight c linenos=table %}
// Example 3
void increase(long c)
{	
    int *t = (int *)c;
    (*t)++;
}

int main()
{
    int a = 0;
    long b = (long)&a; // a의 주소를 b에 저장

    increase(b);
    printf("%d\n", a); // a == 1
}
{% endhighlight %}

함수의 본체가 좀 더 복잡해졌지만, 함수의 인자 전달 과정은 Example 1과 같아졌다. 포인터가 아닌 long 타입 변수 b가 c로 복사됐을 뿐이다.
포인터 변수를 전달하든 일반 변수를 전달하든 그 과정은 다르지 않다는 것이다. 그저 값을 복사할 뿐이다. 그러므로 모든 경우에 Call-by-value가 된다.

여기서 의문이 생겨야 한다. a는 int 타입인데 어째서 b는 int가 아닌 long 타입인가? 

포인터의 크기는 어떻게 결정되는가?
---
int 타입 변수의 주소이므로 주소도 int 타입이라 생각하는 것은 우리 집이 넓다고 주소도 길어질 것으로 생각하는 것과 동급이다.
우리 집의 평수는 주소의 길이에 영향을 주지 않는다. 그렇다면 주소의 길이를 좌우하는 것은 무엇일까?

아파트 하나에 총 100가구가 존재한다면 아파트 내의 주소는 세 글자로 모두 표현할 수 있다. 1000가구가 존재한다면 네 글자로 모두 표현할 수 있다.
중요한 것은 각 가구의 평수가 아니라 가구 수를 결정하는 아파트 전체의 크기다. 

요즘은 64bit 아키텍처를 기본으로 사용한다. $ 2^{64} $이 아파트(가상 메모리)의 크기인 것이다.
이 아파트에 있는 집의 주소는 8Byte 크기가 된다. 왜냐하면 8Byte로 $ 2^{64} $까지의 값을 전부 표현할 수 있기 때문이다.
혹시나 이해가 안간다면.. 1Byte는 8bit로 8bit*8 = 64bit(8byte)다. 1000가구를 네 글자로 모두 표현 가능한 것과 같은 이치다.

그러므로 각 집의 크기가 int(4Byte), char(1Byte)에 관계 없이 포인터(주소 값)는 8Byte가 된다. Example 3또한 int 타입인 a의 주소 값을 저장하기 위해서
long(8Byte) 타입을 사용한다.

{% highlight c linenos=table %}
// Example 4
int main()
{
    int a = 10;
    char b = 'A';

    int *pa = &a;
    char *pb = &b;

    printf("%ld, %ld \n", sizeof(pa), sizeof(pb));
    // output : 8, 8
}
{% endhighlight %}

여기서 다음 의문이다.
포인터의 크기가 늘 같다면 어째서 int의 포인터는 int* 으로 표현하지?
마찬가지로 char의 포인터는 char* 으로 표현한다. 
char의 포인터든 int의 포인터든 크기가 같다면 왜 굳이 타입을 다르게 표시하는 것일까?

포인터에 타입이 존재하는 이유는 무엇인가?
---
포인터가 주소를 가리키는 것이 전부라면 타입은 필요 없다. 하지만 우리는 가리키는 것뿐만 아니라 접근해서 수정도 한다.
예를 들어 아파트 103호에 아주 큰 가구를 넣고 싶다면 이것은 103호라는 주소만 알아서 될 문제가 아니다. 103호가 하필 3평짜리 원룸이라면 못 넣는다. 
주소에 접근해서 뭔가 하려면 대상의 크기도 알아야 하는 것이다. 

{% highlight c linenos=table %}
// Example 5
int main()
{
    int a = 10;
    int *pa = &a;

    printf("%d\n", *pa); // output : 4
}
{% endhighlight %}

Example 5는 잘 동작한다. 포인터의 타입에 int를 명시해서 메모리에 접근할 때 int 크기만큼 읽거나 쓰는 것이다.
아파트의 예로 돌아가면 int는 평수를 의미한다.

자 그럼, pa를 int가 아닌 char 포인터 타입으로 바꾸면 어떻게 될까?
{% highlight c linenos=table %}
// Example 6
int main()
{
    int a = 512;
    char *pa = &a;

    printf("%d\n", *pa); // output : 0
}
{% endhighlight %}

출력 결과는 0이다. 만약 예상했다면 더는 글을 읽을 필요가 없다. int 타입인 a는 4Byte의 공간을 점유하고 있다.
하지만 pa의 타입이 char이므로 char의 크기인 1Byte만큼만 읽는다. 주소는 정확한데 평수를 틀린 것이다.

실제 a의 값은 메모리상에
00000000 00000000 00000001 00000000 로 표현된다. (총 32bit(4Byte)다.)
pa는 여기서 가장 오른쪽 8bit(1Byte)만 읽었기 때문에 0이 나온 것이다.

정리하자면 **포인터의 타입은 몇 바이트나 읽고 쓸 것인가를 표기하는 것이다. 포인터 자체의 크기와는 무관하다.**

이것을 이용하면 int 타입을 크기가 4인 char 타입의 배열로 쓸 수도 있다.

{% highlight c linenos=table %}
// Example 7
int main()
{
    int a;
    char* b = (char*)&a;
    b[0] = 'a';
    b[1] = 'b';
    b[2] = 'c';
    b[3] = 'd';

    for (int i = 0; i < 4; i++)
    {
        printf("%c \n", b[i]);
    }

    return 0;
}
{% endhighlight %}

그렇다면 void*는 뭘까?
---
포인터 타입 중에는 void 타입이 존재한다. 포인터에서 타입은 대상의 크기를 의미한다. 하지만 대상의 크기를 모른다면?

그런 경우가 존재한다. 동적 할당 함수인 malloc을 사용하면 void* 타입을 반환한다. malloc 함수는 입력으로 크기만 받기 때문에
입력값 40이 char 타입 40개인지 int 타입 10개인지 알 수가 없다. 그러므로 사용자가 원하는 만큼 공간을 할당하고 void* 타입을 반환해서 사용자에게 직접 타입을 결정하게 한다.

{% highlight c linenos=table %}
// Example 8
int main()
{
    int *pa = (int *)malloc(40); 
    char *pb = (char *)malloc(40); 
}
{% endhighlight %}

정리
---
포인터의 크기는 어드레스 버스의 크기(32bit or 64bit)와 같다. 포인터의 타입은 주소에 접근한 뒤 얼마만큼 읽거나 쓸 것인지를 의미한다.
malloc 같은 경우 타입을 유추할 수 없으므로 void*를 반환한다. 또한, 크기 정보가 없으므로 읽거나 쓸 수 없다.

아 참, 여기까지 왔으면 자연스레 이해했겠지만 포인터는 메모리를 절약하는 용도로도 사용할 수 있다. 하나의 원본을 가지고 키를 전달하는 방식이기 때문에
늘 작은 크기의 복사만 일어난다. 큰 데이터를 전달할 때도 포인터에 주솟값을 넣은 뒤, 포인터만 복사하면 된다.

하지만 큰 데이터라도 복사를 해야만 할 때도 존재한다. 예를 들어 원본을 보존하면서 복사본을 수정하는 경우도 빈번하다. 경우의 수는 아주 많다.
그러므로 포인터의 본질을 파악하고 상황에 맞게 사용하는 것이 중요하다.


남아 있는 문제
---
어드레스 버스의 크기가 32bit라면 포인터의 크기도 32bit(4Byte)다. 4byte로는 약 43억(4G)까지 표현할 수 있다.
32bit 컴퓨터에서 4GB RAM까지만 인식하는 것도 같은 이유다. 이제 새로운 문제를 생각해보자.

**2GB RAM이 장착된 32bit 컴퓨터가 있다고 가정하자.**
1. 포인터로 2GB 이상의 영역을 참조하면 어떻게 될까?
2. 실행 중인 모든 프로그램이 RAM에 올라간다. 각 프로그램은 이 영역을 어떻게 나눠서 사용할까?
3. 같은 영역을 나눠서 사용한다면 포인터는 다른 프로그램의 변수를 참조할 수 있을까?

이 문제들은 Virtual Memory(가상 메모리)를 도입해서 해결한다. 시간이 되면 가상 메모리에 대한 글도 올리겠다.
