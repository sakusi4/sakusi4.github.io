왜 필요한가?
---
알고리즘 문제를 풀 때나 필요한 것이 아니냐고 생각할 수 있지만, 복잡도 개념을 가지고 있으면 항상 일관된 기준을 가지고 코드를 평가할 수 있다.
다른 개발자들과 코드를 보며 얘기할 때도 효율적이다.

예를 들어 복잡도 개념을 모르는 상태라면 이런 상황을 생각할 수 있다.


A: 100만 개의 원소가 무작위로 들어 있는 배열에서 어떤 항목을 찾아야 하는데, 정렬을 써야하나?

B: 음.. 정렬을 하지 않고 찾으면 순서대로 모두 확인해야 되니까 최대 100만번의 비교가 필요하겠네.. 뭔가 엄청 많다.

A: 퀵 정렬이 되게 빠르대, 정렬된 상태면 이진 탐색을 사용할 수 있는데 데이터가 엄청나게 많아도 순식간에 찾는대

B: 그럼 순서대로 찾는 방법과 정렬 후에 이진 탐색을 하는 방법을 다 해보고 시간을 재볼까?

A: ㅇㅋㅇㅋ


시간 복잡도의 개념이 있다면 이런 대화는 사실 생기지도 않는다. B가 공부를 해왔다. 다시 대답해보자.


A: 100만 개의 원소가 무작위로 들어 있는 배열에서 어떤 항목을 찾아야 하는데, 정렬을 써야 하나?

B: 정렬 알고리즘은 아무리 빨라도 O(NlogN)이야. 검색은 시작도 못했는데 벌써 순서대로 찾는 O(N)보다 느려.

B: O(NlogN)은 1000000log1000000 = 6000000이고 O(N)은 1000000이잖아. 그냥 순서대로 찾는 게 가장 빠른 방법이야.


당연히 후자의 대화가 명확하며 생산적이다. 첫 대화의 문제는 확실한 근거가 없다는 것이다. 시간을 측정해서 순서대로 하는 방법이 더 빠르다는 것을 알았다고 해도
이게 최선의 방법인지 얼마나 더 빠른 것인지 알 수 없다. 

후자의 경우는 대략적인 연산 시간까지도 제시할 수 있고(연산 횟수와 CPU의 클럭 주파수로 계산이 가능하다) 왜 이 알고리즘이 최선인지도 명확한 근거를 가지고 말할 수 있다.

시간 복잡도 계산 방법
---
간단하다. 입력 크기에 따라서 계산 횟수가 어떻게 변하는지 파악하면 된다.
{% highlight c linenos=table %}
int arr[SIZE];
for(int i=0; i<size(arr); i++) {
    if(arr[i] == 10) {
        return i;
    }
}
{% endhighlight %}

위 코드는 SIZE의 크기가 커지면 연산 횟수가 1차 함수로 증가하므로 O(N)이다. 최악의 경우에 N 번을 찾아야 하므로 O(N)이 아니다. (많이들 착각한다.)
입력 크기에 비례해서 선형적으로 증가하므로 O(N)이다. 반복할 때마다 실행되는 명령어를 생각해보자.
1. i<size(arr)의 비교 연산
2. for문 내부의 arr[i] == 10 비교 연산
3. i++

위 세 가지가 반복할 때마다 실행되는 연산이다. 1, 2, 3번 모두 상수 속도로 실행 된다면 이 코드는 O(N)이 맞다. 하지만 size(arr) 함수가 알고 보니 O(N)의 함수였다면, 이 코드는 O(N^2)이 된다. N 번 반복하면서 각 반복마다 N번의 연산을 하기 때문이다. 이런 함정들을 조심해야 한다. 실제로 size를 반환하는 함수가 O(N)인 경우들이 꽤 있다.


{% highlight c linenos=table %}
int arr[SIZE];
for(int i=0; i<SIZE; i++) {
    for(int j=0; j<SIZE; j++) {
        if(arr[i] == 10) {
            return i;
        }
    }
}
{% endhighlight %}

이 코드는 O(N^2)이다. N 번 반복을 N 번하기 때문이다. SIZE가 10이라면 100번 정도의 연산을 한다는 것을 대략 알 수 있다. 만약 안쪽 반복문의 크기 변수가 다르다면
O(N*M)일 것이다. 보면 알겠지만 결국 반복문이 중요하다.

{% highlight c linenos=table %}
for(int i=0; i<SIZE; i++) {
    for(int j=0; j<SIZE; j++) {
        for(int k=0; k<SIZE; k++){
            if(arr[i] == 10) {
                return i;
            }
        }
    }
}
{% endhighlight %}

이 코드는 말할 것도 없이 O(N^3)이다. 트리 구조나 재귀 함수를 사용하면 입력 크기에 따라 지수 함수 또는 로그 함수로 변하므로 O(A^N)이나 O(lgN)이 나올 수도 있다.
임베디드 개발자라면 이러한 계산 방법은 반드시 숙지하고 있어야 한다. 한 번만 계산해보면 시간이 너무 오래 걸린다거나 메모리를 초과한다는 것을 알 수 있으므로 삽질을 방지할 수 있다.

예를 들어보자.

*상황 1*

*여유 메모리 공간 : 1KB*

*입력 데이터 크기 : 100개*

*사용 알고리즘의 공간 복잡도 : O(N^2)*


안된다. O(N^2)이므로 크기가 10000KB(10KB)가 필요하다. 


*상황 2*

*여유 시간 : 0.1초*

*입력 데이터 크기 : 1억개*

*사용 알고리즘의 시간 복잡도 : O(N)*


안된다. CPU마다 성능 차이가 있지만 보통 1억 번의 연산을 하는데 1초로 잡는다. O(N)이 1억 번이므로 1초가 걸린다. CPU 성능의 오차 범위를 무시할 정도로 차이가 있다.


빅-오 표기법은 정확하지 않다.
---
시간 복잡도를 얻었다면 대략적인 연산 횟수도 알 수 있지만 정확하지 않다. 이 오차는 시간 복잡도의 정의에서 온다. 애초에 컴퓨터마다 계산 능력이 다르므로 정확한 추측이 불가능하다. 그러므로 이런 방법을 사용하는 것이다.
{% highlight c linenos=table %}
int arr[SIZE];
for(int i=0; i<SIZE; i++) {
    // 100만번의 명령어 실행
}
{% endhighlight %}

예를 들어 이런 코드가 있어도 O(N)이다. 100만 번이건 1억 번이건 입력의 크기에 따라서 변하지 않는 상수이므로 O(1)로 생각하고 O(1)을 N 번 반복하므로 이 코드는 O(N)이다.

그렇다면 이 코드에서 'SIZE가 100이면 100번 정도 실행하겠네'라는 추측은 아주 크게 빗나간다. SIZE가 100이면 무려 1억 번을 실행한다.
그러므로 시간 복잡도는 정확하지 않다는 것을 항상 인지해야 한다. 뭐 그래도 이런 극단적인 상황은 없겠지만 말이다.

결론
---
복잡도 계산을 활용해서 내 계획이 터무니없는지 미리 알 수 있다. 연산 횟수도 추측이 가능하므로
극단적인 상황을 미리 계산해보고 내 알고리즘이 모든 상황에서 잘 동작한다는 확신을 얻을 수 있다.

빅-오 표기법의 수학적 정의
---
이 부분은 볼 사람만 보자. 몰라도 상관없다.

$f(N) = O(g(N))$이라고 하는 것은 $N_{0}$ 와 임의의 c를 선택했을 때 모든 $N_{0} \leq\ N$에 대해서 $f(N) \leq\ c * g(N)$을 만족한다.

마지막 코드를 대입해보자. g(N)을 N으로 설정한다. $N_{0}$를 1로 잡고 c를 1억으로 잡자. 1보다 큰 모든 N에 대해서
f(N) <= c*g(N)이 성립하면 O(g(N))으로 표현할 수 있다.

f(N)은 1백만이고 c는 1억이므로 이 식은 모든 1과 같거나 큰 N에 대해서 성립한다. 그러므로 O(N)이다.
