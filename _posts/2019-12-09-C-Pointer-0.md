포인터는 어렵기로 유명하지만 실제로는 모호한 부분이 없기 때문에 굉장히 단순하다.

### 포인터를 왜 사용해야 하는 걸까?

포인터를 처음 배우게 되면 그 용도에 대해 의문을 가지게 된다. 뭐 어찌 보면 자연스러운 것이다. 포인터가 유용한 것은 항상 4Byte라는 것이다. (32bit 환경에서) 왜 4Byte일까? 한 번 고민해 본 적이 있는가? 이는 아래에서 설명하는 가상 메모리 공간이 4GB이기 때문이다. 4GB라는 값은 2^32에서 나온 값이다. 32bit 환경이라는 것은 2^32만큼의 주소 공간을 가진다는 것이다. 그러므로 32bit 시스템에서 4GB 이상의 램을 사용하지 못하는 것이다. 그렇다면 4Byte가 표현할 수 있는 최댓값은 얼마인가? 똑같이 2^32다. 4Byte 면 가상 메모리의 주소를 모두 가리키는 데 있어서 문제가 없기 때문에 포인터는 4Byte인 것이다. 

이게 어떠한 점에서 유용한 것일까? 아주 큰 구조체를 인자로 전달한다고 치자. 이는 상당히 부담스러운 연산이 될 것이다. 하지만 해당 구조체의 주솟값 하나를 던져주게 되면 구조체 전체를 전달한 것과 같은 효과를 낼 수 있다. 그리고 복사본이 아니라 원본에 접근할 수 있다는 것이 중요하다. 친구에게 '우리 집에 가서 책 하나만 가져다줘~'라고 말하면서 필요한 것은 집 열쇠를 주는 것이다. 집이 얼마나 크든 간에 열쇠 하나로 집에 있는 모든 데이터에 접근이 가능한 것이다. 똑같은 집을 지어서 들어가 봐야 원했던 책은 존재하지 않는다.
 
### 변수는 어떻게 저장되는가?

주로 처음 변수를 배울 때 아래와 같은 프로그램을 만들어 보면서 배우게 된다.

{% highlight cpp %}
#include <stdio.h>

int main()
{
	int a = 10;
	int b = 20;
	int sum = a + b;
	
	printf("%d",sum);

	return 0;
}
{% endhighlight %}

이 정도 수준의 예제만 만들어 봐도 이후에 변수를 사용하는 프로그램을 만드는데 지장이 없다. 하지만 이렇게만 배우고 넘어갔기 때문에 포인터가 어려운 것이다. 메모리의 개념은 포인터에서 처음 등장하는 것이 아니다. Hello World 분석 포스팅에서 볼 수 있듯이 우리는 Hello World를 출력할 때부터 계속해서 메모리를 사용해왔다. 그렇다면 메모리는 어떻게 생겨 먹었나?

![0](/assets/images/C-Pointer-0/0.jpg)

지금 당창 본체를 열어봐도 확인할 수 있고 게임을 좋아하는 사람들은 다 알고 있을 것이다. RAM이라고 하는 메모리다. 프로그래밍을 공부하다 보면 소프트웨어만 보기 때문에 하드웨어에 대해 생각하지 않게 된다. 하지만 우리가 대화하는 상대는 컴퓨터이고 이는 하드웨어다. 소프트웨어와 하드웨어를 분리하지 말고 동시에 생각하는 습관을 기르도록 하자. 자 그럼 RAM에 대해서 조금만 알아보자. 간단하게 몇 가지만 나열해보겠다.

**1. 비활성 메모리로 전원이 끊기면 메모리의 내용이 유지되지 않는다.**
**2. 우리가 흔히 저장소로 사용하는 HDD, SSD보다 빠른 속도를 가지고 있다.**

아래 사진은 메모리 계층이다. RAM이 하드 드라이브 위에 위치하는 것을 볼 수 있다. 하지만 크기는 작아졌는데 알다시피 RAM보다 HDD의 용량이 더 크다. 이 메모리 계층의 특징은 위로 갈수록 용량이 작아지면서 빨라진다는 것이다.

![1](/assets/images/C-Pointer-0/1.jpg)

정확한 설명을 위해서는 가상 메모리가 등장하게 되고 페이징과 세그먼테이션과 같은 내용도 필요하다. 하지만 이는 지금 레벨에서 필요한 내용은 아니고 나중에 컴퓨터 구조를 공부하게 되면 익히길 바란다. 지금은 자세한 내용은 집어치우고 전체적인 메모리 구조를 머릿속으로 그릴 수 있도록 추상적으로 설명하도록 하겠다.

지금 컴퓨터에 장착되어 있는 RAM의 용량은 어떻게 되는가? 필자는 16GB의 램을 사용하고 있다. 아마 대부분이 4~8GB 정도를 사용할 것으로 생각된다. 하지만 이도 비교적 최근의 일이다. 어릴 때 전자랜드에서 200만 원을 주고 샀던 내 컴퓨터의 메모리는 512MB밖에 되지 않았다. 지금은 말도 안 되는 용량이지만 당시에는 나름 준수한 메모리 용량이었다고 볼 수 있다. 그럼 이런 의문이 든다. 과거와 지금 프로그램이 사용하는 메모리의 크기가 같을까? 

32bit 환경이라면 예전이나 지금이나 프로그램에서 사용하는 메모리의 크기는 4GB로 동일하다. 현재 16GB의 메모리를 사용하는 필자의 컴퓨터에서 프로그램을 작성해도 프로그램의 크기는 4GB로 동일하다. 64bit 프로그램을 작성하면 얘기가 달라지지만 지금은 논외로 하자. 어떻게 모든 프로그램이 4GB를 사용하는지에 대해서는 지금의 주제와 너무 동떨어지기 때문에 다음에 보기로 하고 지금은 이 4GB 영역이 어떻게 구성되는지 확인해보자.

![2](/assets/images/C-Pointer-0/2.png)

가상 메모리의 구조다. 프로그램마다 제공되는 4GB의 메모리 영역을 가상 메모리라고 한다. 2GB는 커널에서 사용하고 사용자는 주로 2GB를 사용한다. 이 공간이 바로 우리 프로그램의 주 무대라고 할 수 있다. 위 사진에서 오른쪽에 살색으로 표시된 영역에 우리가 작성한 코드와 데이터들이 저장되는 것이다. 결과적으로 하고 싶은 말은 우리의 프로그램은 RAM을 사용하며 모든 프로그램에서 동일한 크기의 메모리와 다른 프로그램들과 독립적인 메모리를 사용하기 위해서 가상 메모리를 사용한다. 그럼 이제 실제로 변수가 저장되는 것을 눈으로 확인해보자. 간단한 코드를 작성해보고 어셈블리를 확인해보겠다.

{% highlight cpp %}
#include <stdio.h>

int main()
{
	int a = 10;
	printf("%d\n", &a);
}
{% endhighlight %}

![3](/assets/images/C-Pointer-0/3.png)

먼저 가장 왼쪽에. text라는 것이 보이는가? 위의 메모리 이미지에 살색 부분 메모리 영역을 보면. text에 해당하는 부분이다. 이 부분은 작성한 코드가 있는 영역이다. 이 프로그램에서는 한 개의 변수가 사용되는데 어디에 저장되는지 확인해보자.
  
![4](/assets/images/C-Pointer-0/4.png)

0012FF78 주소에 0A(10진수로 10)가 저장되어 있는 것을 확인할 수 있다. 이처럼 우리가 변수를 선언하고 사용하게 되면 메모리 공간을 할당받게 된다. 이제 이 변수를 사용하기 위해서는 어떻게 해야 할까? 주소에 접근하는 것이다. 하지만 주소에 직접적으로 접근하는 것은 위험한 행위고 직관적이지도 못하다. 그렇기 때문에 우리는 변수명이라는 것으로 메모리 공간에 일종의 별명을 붙여서 변수의 이름으로 메모리 공간을 찾아간다.

위 개념의 연장선상에서 포인터는 전혀 어렵지 않다. 변수에 어떤 값을 대입한다는 것은 변수명이 의미하는 메모리 공간 안에 값을 쓰겠다는 의미다. 포인터에 어떤 값을 대입한다는 것은 변수명이 의미하는 메모리 공간의 주소를 값으로써 쓰겠다는 것이다.

{% highlight cpp %}
#include <stdio.h>

int main()
{
	int a = 10;
	int* pa = &a;

	printf("%d\n", &a);
	printf("%d\n", pa);
}
{% endhighlight %}

그렇기 때문에 위 두 개의 출력 결과는 완전히 동일하다. 헷갈리면 안 되는 것은 포인터도 변수다. 변수지만 가지는 값이 메모리의 주소라는 것이다. 그렇기 때문에 가지고 있는 주솟값을 이용해서 해당 메모리의 내용에 접근이 가능한 것이다. 포인터도 변수라는 것은 이 포인터의 이름도 어떤 메모리 주소를 의미한다는 것이다.

![5](/assets/images/C-Pointer-0/5.png)

위 사진을 자세히 보자. 인텔 CPU는 리틀 엔디안 이라는 방식을 사용해서 값이 거꾸로 저장되는 특징이 있다. 왼쪽의 4Byte를 읽으면 0012FF78이라는 값이 된다. 그리고 아까 확인했듯이 0012FF78의 위치에는 0A(10) 값이 저장돼 있다. 왼쪽의 4Byte는 포인터 변수 오른쪽의 4Byte는 일반 변수의 값이다. 잠깐 정리를 해보겠다.
- **int형 변수 a는 0012FF78 주소에 4Byte 크기의 메모리를 할당받아서 10이라는 값으로 채워져있다.**
- **int형 변수를 가리키는 포인터 pa는 0012FF74 주소에 4Byte 크기의 메모리를 할당받아서 0012FF78이라는 값으로 채워져있다.**

이처럼 포인터 자체도 메모리 공간을 할당받기 때문에 존재하는 위치가 있고 이 위치를 또 다른 포인터가 가리킬 수 있다. 포인터가 포인터를 가리키는 모양새가 되는데 이를 이중 포인터, 더블 포인터라고 한다. 포인터도 변수라는 것을 인지하고 있다면 헷갈릴 내용이 아니다. 그럼 간단하게 변수의 값을 교환해주는 프로그램을 작성해서 확인해보자.

{% highlight cpp %}
#include <stdio.h>

int swap(int* pa, int* pb)
{
	int temp = *pa;
	*pa = *pb;
	*pb = temp;
}
int main()
{
	int a = 10;
	int b = 20;
	
	swap(&a, &b);

	printf("a : %d, b : %d\n", a, b);
}
{% endhighlight %}

![6](/assets/images/C-Pointer-0/6.png)

main 함수의 어셈블리다. 두 개의 변수를 사용하고 있기 때문에 스택에 -8을 해줌으로 공간을 확보한 뒤에 0A(10진수로 10), 14(10진수로 20)을 할당해준다. 그리고 LEA 명령어로 주솟값을 받아 온 뒤 스택에 PUSH 해서 swap 함수로 인자를 전달한다. 스택의 상태를 한번 확인해보자.

![7](/assets/images/C-Pointer-0/7.png)

주솟값까지 자세히 보자. 아래 두 개는 main 함수의 변수다 0012FF74 위치에 14라는 값이 쓰여있고 0012FF78에 0A라는 값이 쓰여있다. 10진수로 각각 10과 20이다. 그리고 위쪽 두 개는 함수에 전달하는 인자인데 14와 0A 값을 전달하는 것이 아니라 14와 0A가 위치하는 주소를 전달하는 것을 볼 수 있다.

![8](/assets/images/C-Pointer-0/8.png)

함수의 대부분이 MOV 명령어로 채워져있다. 데이터들은 이미 스택에 저장되어 있기 때문에 EBP 명령어에 값을 더해줌으로써 원하는 값에 접근이 가능하다. EBP+8에는 10이 들어가 있고 EBP+C에는 20이 들어가 있다. swap 함수에 전달된 주솟값으로 main 함수의 스택 영역에 존재하는 두 개의 변수에 접근하고 있다.

그리고 ECX 임시 변수를 사용해 값을 교환하는 것이 끝이다. 이 모든 일련의 작업이 주솟값에 접근해서 이루어지는 것이므로 main 함수에서 값을 출력해보면 값이 바뀐 것을 확인할 수 있다. 만약 변수의 값을 전달해서 교환했다면 메모리 공간의 복사가 일어난 뒤 값의 교환이 생기므로 main 함수에는 아무런 영향이 생기지 않았을 것이다.

주솟값을 사용하는 것이 포인터의 전부다. 알고 보면 간단한 내용을 이렇게 장황하게 설명한 것은 포인터를 단순히 문법적인 측면에서만 끝낼 것이 아니라 컴퓨터와 메모리 구조와 같은 큰 그림을 그리며 생각할 수 있는 것이 더 좋다.
